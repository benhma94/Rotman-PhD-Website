Three Approaches for the PhD Association Site

All use GitHub Pages + a client-side password gate. Pick the build approach that fits best.

A

Plain HTML / CSS / JS

Hand-written HTML pages, no build tools. You edit the HTML directly to update content.

Pros

  • Zero setup — push files, site is live
  • No dependencies, never breaks
  • Full control over every pixel

Cons

  • Content and code are mixed — updating a notice means editing HTML
  • Harder for non-technical future admins
B

Jekyll (GitHub Pages native) ⭐ Recommended

GitHub Pages has Jekyll built-in — push markdown/YAML files and GitHub builds the site automatically. No CI/CD setup needed.

Pros

  • Content lives in simple YAML/markdown files (easy for future admins)
  • GitHub builds it for free — zero extra tooling
  • Future admins can edit content via GitHub's web editor (no code knowledge needed)
  • Clean separation: change a resource link without touching HTML

Cons

  • Small upfront learning curve for the data file format
  • Slightly slower GitHub Pages build (~1–2 min after a push)
C

React / Next.js (static export)

Modern component-based framework with a GitHub Actions build step to deploy to Pages.

Pros

  • Easiest to add interactive features later (polls, dynamic filtering)
  • Great ecosystem of UI libraries

Cons

  • Overkill for a resources + calendar site
  • Requires GitHub Actions setup for deployment
  • Node.js dependency management can be a maintenance burden
